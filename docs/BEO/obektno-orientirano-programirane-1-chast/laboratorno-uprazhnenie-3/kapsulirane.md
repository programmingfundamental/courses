# Капсулиране (Encapsulation)

**Капсулирането** е един от четирите основни принципа на обектно-ориентираното програмиране. То представлява процес на „опаковане“ на данните (атрибутите) и методите, които работят с тях, в единна единица – **класа**.

Основната цел на капсулирането е да се ограничи директният достъп до вътрешното състояние на обекта и да се осигури контролиран интерфейс за взаимодействие с него. Това помага за предотвратяване на неволни или неправилни модификации на данните.

### Как работи капсулирането?
1. **Скриване на данни (Data Hiding):** Всички атрибути на класа се декларират като частни (`private`). По този начин те стават невидими за външни класове.
2. **Публичен интерфейс:** Достъпът до тези атрибути се осъществява единствено чрез публични (`public`) методи – **getters** (за четене) и **setters** (за запис).

## Примери за капсулиране

### 1. Модифицируеми обекти (Mutable Objects)
При тях състоянието може да се променя след създаването на обекта чрез използване на **setter методи**.

```java
class Person {
    private String name;
    private int age;

    // Гетър за името
    public String getName() {
        return name;
    }

    // Сетър за името - позволява промяна
    public void setName(String name) {
        this.name = name;
    }

    // Сетър за годините с вградена валидация
    public void setAge(int age) {
        if (age > 0) {
            this.age = age;
        }
    }
}
```

### 2. Немодифицируеми обекти (Immutable Objects)
При тях състоянието се задава веднъж при създаването (чрез конструктор) и не може да бъде променено по-късно. Това се постига чрез:
* Деклариране на полетата като `private final`.
* Предоставяне само на **getter методи** (без сетери).

```java
class ImmutablePerson {
    private final String name; // 'final' гарантира, че стойността няма да се промени

    public ImmutablePerson(String name) {
        this.name = name;
    }

    public String getName() {
        return name;
    }
}
```

## Предимства на капсулирането

* **Контрол и валидация:** Чрез setter методите можем да контролираме какви стойности се присвояват на нашите променливи (например да предотвратим задаването на отрицателна възраст).
* **Гъвкавост:** Можем да направим даден атрибут „само за четене“ (readonly), като не дефинираме setter метод за него.
* **Скриване на сложността:** Потребителят на класа не е необходимо да знае как точно се съхраняват данните вътрешно. Той използва само методите, предоставени от публичния интерфейс.
* **Лесна поддръжка:** Ако решим да променим името или типа на някоя вътрешна променлива, ще трябва да актуализираме само методите вътре в самия клас, без да засягаме останалата част от програмния код, който го използва.
* **Сигурност:** Предпазва обекта от преминаване в невалидно състояние.
